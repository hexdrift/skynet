§Overview

Skynet runs as two pods (frontend + backend) plus an in-cluster migration Job, fronted by the OpenShift router. Every external dependency below resolves inside the air gap — your job is to point the chart at the right internal hosts.

Skynet runs as a backend API container plus a bundled pgvector Postgres on a single host via Docker Compose — the fastest way to stand the app up inside the air gap. Every external dependency below still resolves internally; you point the backend's .env at the right internal hosts instead of a Helm values file. Start at Single-Host (Docker Compose).

§Single-Host (Docker Compose)

The fastest path inside the air gap: one backend API container plus a bundled pgvector Postgres, defined in backend/docker-compose.yml. No Helm, no cluster, no separate migration Job — the backend creates its own schema on first boot. This brings up the backend service and its database; the Next.js frontend and SSO are a separate concern not covered by this compose file.

Prerequisites

Docker Engine with the Compose v2 plugin (docker compose version). Everything else — the pgvector/pgvector:pg16 image and the backend image's build dependencies — comes from your internal mirrors via the build args below, so nothing touches Docker Hub or public PyPI.

1 · Configure .env

Copy the template and change the placeholders (see Secrets You Must Change):

cd backend
cp .env.example .env            # then edit .envcd backend
copy .env.example .env          REM then edit .env

At minimum set POSTGRES_PASSWORD (the bundled DB refuses to start without it), point OPENAI_API_BASE and the agent *_BASE_URL at your internal gateway, and set OPENAI_API_KEY to the token it expects. The exact model-id prefix and base-URL suffix depend on your gateway flavor — pick it with the LLM gateway selector above and copy the matching block from Runtime Egress. Compose passes .env to the backend verbatim and composes REMOTE_DB_URL for you against the bundled db service, so you never hand-write a database URL.

2 · Point the build + image pulls at internal mirrors

Set these before up so neither the Postgres image nor the backend build reaches a public registry:

export REGISTRY_PREFIX=artifactory.example.com/docker-remote
export DEBIAN_MIRROR=https://artifactory.example.com/debian-remote
export PIP_INDEX_URL=https://artifactory.example.com/api/pypi/pypi-remote/simpleset REGISTRY_PREFIX=artifactory.example.com/docker-remote
set DEBIAN_MIRROR=https://artifactory.example.com/debian-remote
set PIP_INDEX_URL=https://artifactory.example.com/api/pypi/pypi-remote/simple

REGISTRY_PREFIX prefixes the pgvector/pgvector:pg16 image and the backend base image; DEBIAN_MIRROR / PIP_INDEX_URL feed the backend build. The image installs from uv.lock via uv by default — the Installer · uv selection. To build the pip path instead, add PKG_MANAGER: pip under services.api.build.args in docker-compose.yml; the compose file does not forward it by default.

3 · Bring it up

docker compose up --build

Add -d to run detached. On first connect the backend bootstraps its full schema under a Postgres advisory lock (Base.metadata.create_all), so a fresh database needs no manual migration. When SEARCH_BACKEND=semantic it also runs CREATE EXTENSION vector and builds the HNSW indexes automatically — the bundled pgvector/pgvector image's local role has the privilege.

4 · Smoke test

curl -f http://localhost:8000/healthcurl.exe -f http://localhost:8000/health

The API listens on host port 8000, Postgres on 5432. The backend serves plain HTTP — put your own reverse proxy in front of :8000 if you need TLS; Compose terminates none.

Search backend on Compose

  • Lexical (default) and semantic both work out of the box — the bundled image is pgvector/pgvector, so SEARCH_BACKEND=semantic only needs an internal EMBEDDINGS_BASE_URL / EMBEDDINGS_MODEL in .env.
  • BM25 needs the pg_search extension, which is not in pgvector/pgvector:pg16; there SEARCH_BACKEND=bm25 falls back to ILIKE. Swap in a ParadeDB image for true BM25.

Upgrades & persistence

Data lives in the named volume skynet-pgdata. The first-boot bootstrap only creates missing tables (current as of the running image) — it records no Alembic version and never alters an existing schema. To track it for future delta migrations, stamp it once at the head the image already built, then upgrade through new revisions when you move to a newer image:

# one-time: adopt the auto-created schema into Alembic
docker compose run --rm api alembic stamp head

# later, after pulling a newer backend image:
docker compose run --rm api alembic upgrade head
docker compose up --build -d

Take a pg_dump (or snapshot the skynet-pgdata volume) before every upgrade — some revisions are destructive, and the pre-upgrade dump is the only real recovery path.

§Secrets You Must Change

Several shipped values are placeholders that work in dev and fail — or open a hole — in production. Change every one before exposing the app. Where they live depends on the deploy target: .env for Compose, or the Kubernetes Secrets for the Helm path.

ValueShips asSet it to / failure mode
POSTGRES_PASSWORDreplace-meA strong random string. Compose's db refuses to start without it; on Helm it backs the skynet-db-password Secret.
OPENAI_API_KEYreplace-meThe bearer token your internal LLM gateway expects (any non-empty placeholder like not-needed for an unauthenticated gateway — an empty value can break client init). Doubles as the embeddings bearer unless EMBEDDINGS_API_KEY is set.
BACKEND_AUTH_SECRETchange-me-in-productionOne openssl rand -base64 32, byte-identical in the backend and frontend. Mismatch ⇒ every admin API call returns 401; unset on the backend ⇒ 500 auth.not_configured. Rotating it needs a coordinated restart of both.
AUTH_SECRETchange-me-in-productionA separate openssl rand -base64 32 for NextAuth session signing (frontend, from frontend/.env.example). Generate your own; don't ship the placeholder.
Security · do not ship dev login

When SSO is left unconfigured — any of AUTH_SSO_ISSUER, AUTH_SSO_CLIENT_ID, or AUTH_SSO_CLIENT_SECRET missing — the frontend falls back to a passwordless “Dev Login”: any username signs in. In that state AUTH_ADMINS defaults to admin, so whoever logs in as user admin is auto-granted admin. For any real deployment configure OIDC (see ADFS / OIDC Setup), set AUTH_ADMIN_GROUPS / backend ADMIN_GROUPS to a real IdP group, and keep AUTH_ADMINS / ADMIN_USERNAMES as a named break-glass list only.

§Crossing the Air Gap: Images & Checksums

There are two ways to get images across, and they have very different connectivity requirements.

Path A — build on a connected host, then carry tarballs across

Build on a host that still has internet (or reaches public mirrors), export each image to a tar, carry the tars across on physical media, then docker load and push to the internal registry. This is the path for a fully disconnected deploy environment.

# On the connected build host:
docker save artifactory.example.com/skynet/skynet/backend:2026.04.30 \
  | gzip > backend-2026.04.30.tar.gz
docker save artifactory.example.com/skynet/skynet/frontend:2026.04.30 \
  | gzip > frontend-2026.04.30.tar.gz
# skopeo is an alternative that copies straight into an OCI archive:
# skopeo copy docker-daemon:...:2026.04.30 oci-archive:backend.tar:2026.04.30

# After carrying across and loading on the internal side:
gunzip -c backend-2026.04.30.tar.gz | docker load
gunzip -c frontend-2026.04.30.tar.gz | docker load
docker push artifactory.example.com/skynet/skynet/backend:2026.04.30
docker push artifactory.example.com/skynet/skynet/frontend:2026.04.30REM On the connected build host (PowerShell):
docker save artifactory.example.com/skynet/skynet/backend:2026.04.30 -o backend-2026.04.30.tar
docker save artifactory.example.com/skynet/skynet/frontend:2026.04.30 -o frontend-2026.04.30.tar

REM After carrying across and loading on the internal side:
docker load -i backend-2026.04.30.tar
docker load -i frontend-2026.04.30.tar
docker push artifactory.example.com/skynet/skynet/backend:2026.04.30
docker push artifactory.example.com/skynet/skynet/frontend:2026.04.30

Path B — in-place build-images on the internal side

The build-images command builds directly on the internal build host. This is not a fully disconnected path: the build host must reach internal PyPI / npm / Debian / Docker Hub mirrors live (see Build Images). Use it only when those mirrors are reachable from the build host. Pin base images by digest in the internal registry so a mirror retag cannot change what you build.

Checksums

Emit a SHA256SUMS for every bundle artifact (image tars, source archive, Helm chart, CA bundle) on the source side and verify it on the destination side before you load or install anything.

# Source side — emit:
sha256sum backend-2026.04.30.tar.gz frontend-2026.04.30.tar.gz \
  skynet-source.tar.gz skynet-chart.tgz internal-ca-bundle.pem > SHA256SUMS

# Destination side — verify (non-zero exit = mismatch, do not proceed):
sha256sum -c SHA256SUMSREM Source side — emit (PowerShell):
Get-FileHash -Algorithm SHA256 backend-2026.04.30.tar,frontend-2026.04.30.tar,skynet-source.zip,skynet-chart.tgz,internal-ca-bundle.pem |
  Format-List > SHA256SUMS.txt

REM Destination side — re-run Get-FileHash and compare each value to SHA256SUMS.txt.
Get-FileHash -Algorithm SHA256 backend-2026.04.30.tar

§External Postgres Prerequisites (search backend)

Which Postgres extension — if any — the database needs is decided entirely by SEARCH_BACKEND in the backend env block. The pre-install migrate Job and the application pods read the same value:

  • SEARCH_BACKEND=lexical (default) — no extension. Explore search uses ILIKE substring matching on a vanilla Postgres. The baseline migration skips CREATE EXTENSION vector and the job_embeddings vector schema, so a database with no extension privilege installs cleanly. Recommended when you can't add extensions in the air gap.
  • SEARCH_BACKEND=bm25 — needs pg_search. Ranked lexical search via the ParadeDB pg_search extension; degrades to ILIKE if it's absent (best-effort, never blocks boot). Still no pgvector.
  • SEARCH_BACKEND=semantic — needs pgvector. Embedding (vector) search. Only this profile makes the baseline migration run CREATE EXTENSION vector, create the vector(512) columns and write job_embeddings; it also needs an internal embedding gateway (EMBEDDINGS_BASE_URL/EMBEDDINGS_MODEL).

Only if SEARCH_BACKEND=semantic: pgvector

For the semantic profile pgvector is a hard prerequisite — the baseline migration runs CREATE EXTENSION IF NOT EXISTS vector and creates vector(512) columns. If you point the chart at an external / managed Postgres (postgres.enabled=false), the database must satisfy both of these before install:

  • The pgvector server extension binary is installed on the database host. CREATE EXTENSION only registers an already-installed extension — it cannot install the binary. On RDS-like managed systems this means pgvector must be on the platform's extension allowlist.
  • The migration role can run CREATE EXTENSION vector (a privileged operation), or a DBA has pre-created the extension and the migration role can see it.

The safest route on a managed DB is to have a DBA pre-run the extension and verify it:

CREATE EXTENSION IF NOT EXISTS vector;
SELECT * FROM pg_extension WHERE extname = 'vector';
TODO · On-premise

The pre-install migrate Job reads SEARCH_BACKEND from the same backend ConfigMap as the pods. With the default lexical (or bm25) the baseline migration skips CREATE EXTENSION vector entirely, so a vanilla Postgres with no extension privilege installs cleanly. Only SEARCH_BACKEND=semantic requires pgvector — and there, because the migration runs as a pre-install hook, a missing extension or insufficient privilege aborts the entire install before any application pod starts.

Optional: pg_search for BM25 lexical ranking

The default deployment runs lexical explore search (no embedding gateway needed). Set SEARCH_BACKEND=bm25 and install the pg_search (ParadeDB) extension on this database to rank that search with true BM25 relevance; without the extension it falls back to ILIKE substring matching. This is best-effort — at startup the backend tries to create the extension and a bm25 index over the jobs corpus, and silently uses ILIKE if it can't (so a missing pg_search never blocks boot, unlike the vector requirement above). To enable BM25, set SEARCH_BACKEND=bm25, install/allowlist pg_search on the managed Postgres, and confirm:

CREATE EXTENSION IF NOT EXISTS pg_search;
SELECT * FROM pg_extension WHERE extname = 'pg_search';

Backend logs show BM25 lexical search enabled (pg_search) when it activates.

§Pre-Transfer Checklist

Run this before moving the repo or release bundle into the air gap:

scripts/airgap_migrate.sh check
scripts/airgap_migrate.sh validate-migrations
scripts/airgap_migrate.sh todos          # every URL/secret/CIDR you must changescripts\airgap_migrate.bat check
scripts\airgap_migrate.bat validate-migrations
scripts\airgap_migrate.bat todos         REM every URL/secret/CIDR you must change

Confirm

  • frontend/package-lock.json is present.
  • backend/uv.lock and backend/requirements.txt are present (the uv and pip installer inputs).
  • backend/alembic/versions/ is present.
  • migration.sql (from validate-migrations) reflects the schema you expect.
  • Your internal OpenAI-compatible embedding endpoint is ready and returns at least 512 dimensions (the embeddings_dim default) for EMBEDDING_MODEL (placeholder jina-embeddings-v4) — semantic search reaches it at runtime.
  • Backend, frontend, and pgvector images are available in the internal registry you will reference from Helm.
  • If your IdP, LLM gateway, or Postgres uses a private CA, a Kubernetes Secret containing the CA bundle is ready.

Recommended release bundle contents

  • Git checkout or source archive.
  • backend image tar or internal registry image.
  • frontend image tar or internal registry image.
  • pgvector/pgvector:pg16 mirror, unless using an external managed DB.
  • Helm chart under deploy/helm/skynet.
  • scripts/airgap.env.example — the fill-in worksheet for every site-specific value (registry, hosts, IdP, gateway, DB, egress CIDRs).
  • Secrets material handled by your secret-management process, not committed into the bundle.

Mirror notes

  • Container base images: pass REGISTRY_PREFIX for backend and BASE_IMAGE for frontend. You can also pre-bake internal skynet/python-base and skynet/node-base images.

§Internal LDAP / Active Directory User Search

ADFS handles sign-in. For the admin "set a quota for any network user" flow, the backend also needs to search Active Directory so admins can type a fragment and pick a real domain user from autocomplete.

The lookup goes through backend/core/api/directory_client.py. When LDAP is not configured the backend uses NullDirectoryClient: the autocomplete combobox falls back to usernames already seen in the local job + override tables, which is enough for any user who has signed in or has an existing override. Admins can still type a brand-new username manually and save a quota for them — the field accepts free text. LDAP only adds discovery of users who have not yet touched the system.

The Python client uses the ldap3 package (already in backend/pyproject.toml, backend/uv.lock, and backend/requirements.txt, so it ships in the airgap install bundle without extra wheels).

    AD_LDAP_SEARCH_BASE: "DC=example,DC=internal"

    AD_LDAP_USER_FILTER: "(&(objectCategory=person)(objectClass=user))"
    AD_LDAP_USERNAME_ATTR: "sAMAccountName"
  secrets:
    existingSecret: "skynet-backend-secrets"   # must also contain AD_LDAP_BIND_PASSWORD

Network + TLS

  • The backend pod must reach AD_LDAP_URL (typically TCP/636 for ldaps://, TCP/389 for ldap://). Add the AD controller CIDR/port to networkPolicy.ldapEgress in the generated values; that slot exists precisely so the LDAP allowlist sits next to the LLM allowlist instead of competing with the broader egressCidrs list. Set LDAP_EGRESS_CIDR (and optionally LDAP_EGRESS_PORT, default 636) before running scripts/airgap_migrate.sh valuesscripts\airgap_migrate.bat values to emit it automatically.
  • For ldaps:// over a private CA, mount the CA exactly as the OIDC setup does — see Private TLS / CA Bundles. The same SSL_CERT_FILE / REQUESTS_CA_BUNDLE mount that the backend already uses for the LLM gateway and Postgres covers ldap3 as well, since ldap3 honours the Python TLS trust store.

Username consistency

Quotas, audit events, and admin lookups all key on the lowercased username string. Whatever attribute you set as AD_LDAP_USERNAME_ATTR should match the value the frontend stores in the session — that is the claim selected by AUTH_USERNAME_CLAIM (or its fallback chain name → unique_name → upn → preferred_username → email → sub). If you use userPrincipalName in ADFS claims, set AD_LDAP_USERNAME_ATTR=userPrincipalName so autocomplete returns the same string the user signs in as.

§Database Migration

Offline schema review (no DB required)

Before pushing the backend image, you can dump the exact SQL the in-cluster migration Job will execute. This is the path to use when your build host has no Postgres reachable (e.g., sandboxed CI):

scripts/airgap_migrate.sh validate-migrations
# wrote offline migration SQL: ./migration.sqlscripts\airgap_migrate.bat validate-migrations
REM wrote offline migration SQL: .\migration.sql

Under the hood this calls alembic upgrade head --sql against the same alembic/env.py the Job uses, so the offline output is byte-for-byte the DDL the cluster will run. Review ./migration.sql, then proceed to build-images / install.

A few revisions also run a data backfill (per-row reads that branch in Python — e.g. the legacy 0-1 → 0-100 metric rescale). Those cannot be expressed as static SQL, so in the offline dump they appear as a -- ... runs online only marker and are executed for real by the in-cluster Job against a live connection. Their absence from migration.sql is expected; the schema (DDL) is still complete.

In-cluster migration Job

The Helm chart runs a pre-install/pre-upgrade Job:

migration:
  enabled: true
  command: ["alembic", "upgrade", "head"]

The migration Job uses the same backend image and the same REMOTE_DB_URL composition as backend pods. Helm waits for the Job to succeed before starting new backend pods.

First install or upgrade:

scripts/airgap_migrate.sh installscripts\airgap_migrate.bat install

Check migration logs:

kubectl -n skynet logs job/skynet-skynet-migrate

If you are adopting a database that already has the baseline schema created outside Alembic, do a one-time stamp at the baseline revision so subsequent upgrades only apply the deltas:

helm upgrade --install skynet deploy/helm/skynet \
  -n skynet --create-namespace \
  -f deploy/helm/skynet/values-airgap.generated.yaml \
  --set-json 'migration.command=["alembic","stamp","342f7449be26"]'helm upgrade --install skynet deploy\helm\skynet ^
  -n skynet --create-namespace ^
  -f deploy\helm\skynet\values-airgap.generated.yaml ^
  --set-json "migration.command=[\"alembic\",\"stamp\",\"342f7449be26\"]"

342f7449be26 is the squashed baseline (all jobs, job_*, and quota tables). To adopt a DB that sits at that baseline, stamp it there and then upgrade through every delta: alembic stamp 342f7449be26 followed by alembic upgrade head. The single current head is c1d2e3f4a5b6.

Only run alembic stamp head when alembic current already equals the live head c1d2e3f4a5b6 — i.e. the schema is genuinely fully migrated. Stamping head tells Alembic the DB is current and silently skips every unapplied migration. The job_embeddings.is_private column was added in a1b2c3d4e5f6, roughly 22 revisions below head, so a DB that only has is_private is not at head; stamping head there would skip ~22 migrations and corrupt the schema. Verify the head matches the shipped image with alembic heads (and alembic current against the live DB) from backend/ before deciding.

After a baseline stamp, revert migration.command to ["alembic", "upgrade", "head"] for the next upgrade.

§Backup, Upgrade & Rollback

The schema migration is destructive on some revisions, so the real recovery path is always a fresh dump taken immediately before you change anything.

Mandatory backup before install / upgrade

Take a full pg_dump of the external database before every install or upgrade and keep it until the new release is verified:

pg_dump \
  --host=pgvector.internal --port=5432 \
  --username=skynet --dbname=skynet \
  --format=custom --no-owner \
  --file=skynet-preupgrade-2026.04.30.dumppg_dump ^
  --host=pgvector.internal --port=5432 ^
  --username=skynet --dbname=skynet ^
  --format=custom --no-owner ^
  --file=skynet-preupgrade-2026.04.30.dump

How the pre-upgrade migration Job behaves on failure

The migration runs as a pre-install/pre-upgrade hook. If it fails, Helm aborts the release — but the schema may already be partially applied, because Alembic commits each revision as it goes. The release is rolled back at the Helm level; the database is not. That is exactly why the pre-upgrade pg_dump above is mandatory.

Rolling back

There are two independent layers to roll back — the chart and the schema.

  • Chart: helm rollback <release> <rev> returns the workloads, NetworkPolicies, and config to a previous Helm revision. It does not touch the database schema.
  • Schema: run alembic downgrade as a one-off Job that overrides migration.command, e.g. --set-json 'migration.command=["alembic","downgrade","-1"]'.
helm rollback skynet # to the previous revision; or specify a rev number

# schema downgrade as a one-off Job:
helm upgrade --install skynet deploy/helm/skynet \
  -n skynet \
  -f deploy/helm/skynet/values-airgap.generated.yaml \
  --set-json 'migration.command=["alembic","downgrade","-1"]'helm rollback skynet REM to the previous revision; or specify a rev number

REM schema downgrade as a one-off Job:
helm upgrade --install skynet deploy\helm\skynet ^
  -n skynet ^
  -f deploy\helm\skynet\values-airgap.generated.yaml ^
  --set-json "migration.command=[\"alembic\",\"downgrade\",\"-1\"]"

Connection-pool math

Total DB connections = backend replicas × (DB_POOL_SIZE + DB_POOL_MAX_OVERFLOW). Keep that under the Postgres max_connections (default 200). For example, 3 backend replicas with DB_POOL_SIZE=10 and DB_POOL_MAX_OVERFLOW=10 can open up to 60 connections.

When you scale backend replicas horizontally, enable pgbouncer so the pods share a bounded server-side connection pool instead of each opening their own — otherwise replica count multiplied by pool size races past max_connections and new connections are refused.

§Runtime Egress

Internal LLM gateway

Each model the backend calls is a base URL + model id pair. The *_MODEL id is forwarded verbatim by LiteLLM to the matching *_BASE_URL, so the id must exactly match a model your gateway actually serves.

Base URLModel id
CODE_AGENT_BASE_URLCODE_AGENT_MODEL
GENERALIST_AGENT_BASE_URLGENERALIST_AGENT_MODEL
EMBEDDINGS_BASE_URLEMBEDDINGS_MODEL

Leaving any *_BASE_URL empty falls through to a public OpenRouter default (openrouter/minimax/minimax-m3), which an air-gapped host cannot reach — set every base URL to your internal gateway. The *_MODEL ids must exactly match models your gateway serves; the shipped gpt-5 default is only a placeholder. The embedding model placeholder is jina-embeddings-v4 and must return at least 512 dimensions.

Gateway flavor: id prefix + base-URL suffix

The runtime is gateway-agnostic, but the two strings you hand-type differ per flavor: the model-id prefix and the base-URL suffix. Pick yours with the LLM gateway selector above. In every case repoint OPENAI_API_BASE at the same internal gateway and leave OPENROUTER_API_KEY / ANTHROPIC_API_KEY / FIREWORKS_AI_API_KEY unset — a wrong prefix or a public default 404s (or phones home) the moment egress is blocked.

# Ollama — native API, NO /v1 suffix; the ollama/ prefix is load-bearing

networkPolicy.dbEgressManaged Postgres — a list of {cidr, ports:[5432]}. The migration Job also gets its own pre-install NetworkPolicy hook so it can reach the DB before the backend pods exist.

External Postgres egress

When postgres.enabled=false, you must set networkPolicy.dbEgress. Without it, default-deny blocks both the backend connection pool and the pre-install migration Job from reaching the managed DB, so the install aborts on the migration hook and the backend pods crash-loop. Set DB_EGRESS_CIDR before running values to emit it.

Known risky endpoint by design

  • POST /models/discover accepts a user-supplied base_url so operators can discover models from an internal OpenAI-compatible gateway. In an air gap, enforce egress with NetworkPolicy/firewall rules.

The generated values file sets these slots to placeholders. Tighten every one to exact internal CIDRs.

§No-Phone-Home Checklist

The default config looks deployed but quietly reaches the public internet. Confirm each of these before you cut egress — one missed item keeps a model or summarisation call going out.

  • Both agents default to openrouter/minimax/minimax-m3 (DEFAULT_AGENT_MODEL_ID), which needs egress to openrouter.ai. Override CODE_AGENT_MODEL + CODE_AGENT_BASE_URL and GENERALIST_AGENT_MODEL + GENERALIST_AGENT_BASE_URL to your internal gateway (see the gateway-flavor blocks).
  • Leave EMBEDDINGS_SUMMARY_MODEL empty. Empty, the job-summary step reuses CODE_AGENT_MODEL (already internal). Setting it to a public model id routes summarisation calls outside the air gap.
  • DISCOVER_ALLOW_PRIVATE is the counterintuitive one. If you use POST /models/discover to enumerate models from an internal gateway on a private (RFC1918, e.g. 10.x) address, set DISCOVER_ALLOW_PRIVATE=true — the default false blocks it as SSRF protection, so discovery against your own gateway fails closed. Loopback (Ollama on localhost) always works; cloud-metadata IPs (169.254.169.254) stay blocked regardless.
  • Leave OPENROUTER_API_KEY / ANTHROPIC_API_KEY / FIREWORKS_AI_API_KEY unset. They only matter for public providers; an empty *_BASE_URL paired with one of these set is how the agent (and the grounding scorer) silently reach a public API.
  • Already handled for you: LITELLM_LOCAL_MODEL_COST_MAP=True is baked into the backend image and chart, so LiteLLM uses its bundled cost map instead of fetching it from GitHub on import, and LiteLLM telemetry is off. Don't unset them.

§Private TLS / CA Bundles

If internal endpoints use a private CA, set INTERNAL_CA_SECRET while running scripts/airgap_migrate.sh configurescripts\airgap_migrate.bat configure or values. The generated values file will:

  • mount the CA Secret into backend and frontend pods,
  • set NODE_EXTRA_CA_CERTS for NextAuth/OIDC in the frontend,
  • set SSL_CERT_FILE and REQUESTS_CA_BUNDLE for Python HTTP clients in the backend and migration Job.

Example:

INTERNAL_CA_SECRET=skynet-internal-ca \
INTERNAL_CA_FILENAME=ca-bundle.pem \
scripts/airgap_migrate.sh valuesset INTERNAL_CA_SECRET=skynet-internal-ca
set INTERNAL_CA_FILENAME=ca-bundle.pem
scripts\airgap_migrate.bat values

§Live User Quotas

The default optimization cap is MAX_JOBS_PER_USER=100. Static env values still work for broad defaults:

  • QUOTA_OVERRIDES: JSON map such as {"power@example.com": 500} or {"researcher@example.com": null}.

For day-to-day changes, use the admin tab in the settings modal. Admins can set a numeric quota, set a user to unlimited, or remove a live override so the user falls back to the default/static config.

Deploy the migration once so user_quota_overrides and user_quota_audit_events exist. After that, backend quota enforcement reads the override table on each submission/clone/retry, so updates take effect immediately without a backend restart. The admin tab also shows the latest quota audit events.

§What Is Already Air-Gap Clean

  • Scalar API docs are vendored under backend/core/api/static/scalar/.
  • Fonts are bundled through npm packages, not Google Fonts. Notification emails no longer reference Google Fonts either — they use self-hosted or system-fallback fonts only.
  • Explore search defaults to lexical (SEARCH_BACKEND=lexical) on a vanilla Postgres with no extensions — the migrate Job never runs CREATE EXTENSION, so the default deployment needs neither an embedding gateway nor pgvector. SEARCH_BACKEND=bm25 adds ranked lexical search via pg_search (ILIKE fallback). SEARCH_BACKEND=semantic opts into vector search: it needs pgvector plus an internal OpenAI-compatible EMBEDDINGS_BASE_URL/EMBEDDINGS_MODEL. No embedding model weights are bundled in the repo.
  • LiteLLM telemetry is disabled in backend configuration, and LITELLM_LOCAL_MODEL_COST_MAP pins it to its bundled cost map so it does not fetch the cost map from GitHub on import.
  • Storage is Postgres; there is no S3/GCS/Azure Blob dependency.
  • Auth is Credentials plus generic OIDC; no Google/GitHub OAuth provider is configured.

§Remaining Operator Responsibilities

The repo cannot automate these site-specific pieces:

  • Mirroring/pushing images and package repositories into your internal Artifactory/registry.
  • Creating Kubernetes Secrets and rotating credentials.
  • Supplying exact NetworkPolicy CIDRs for IdP, LLM gateway, embedding gateway, LDAP, and DB.
  • Backing up and restoring Postgres.
  • Choosing internal model IDs exposed by your LLM gateway. The generated values default CODE_AGENT_MODEL and GENERALIST_AGENT_MODEL to gpt-5; change them if your gateway exposes different names. Leaving CODE_AGENT_BASE_URL / GENERALIST_AGENT_BASE_URL empty falls back to a public OpenRouter model that an air-gapped host cannot reach.
  • Semantic search is selected: provision pgvector on the database, choose the internal embedding model exposed by your embedding gateway, and set EMBEDDINGS_BASE_URL. (Lexical / BM25 need none of this.)
